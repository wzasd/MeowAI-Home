# 开发日记 — Inline Queue + Composer 视觉重设计

## 今天做了什么

今天没有继续堆功能，而是给新加的“排队发送”体验补了一层真正的产品设计。

现状已经能排队、强制发送、取消排队，但视觉上还停留在“技术功能条”：

- 队列像一段浅黄色列表
- `排队发送` 和 `强制发送` 更像两个调试按钮
- 用户第一眼看不出“下一条发给谁、内容是什么、是否值得插队”

所以这次把重点放在：

1. 重新定义队列的隐喻
2. 让队列和输入区形成一个整体
3. 产出能直接给前端实现的静态稿

## 设计决策

### 1. 不叫“队列面板”，改成“发件夹”

如果继续做“排队面板”，产品气质会越来越像后台监控。

这次改成：

- 输入框 = 写信台
- 排队区 = 待送信笺 / 发件夹
- 第一条待发消息 = 主卡
- 后续消息 = 压缩信笺

这样用户的心理模型不再是“系统里堆了 2 条”，而是“我已经写好了两封，第一封马上送”。

### 2. 第一条必须被突出

旧设计里每一条都是同权的，导致用户要逐条扫。

新方案里：

- 第一条是 hero card
- 第二条以后退成小行
- 目标猫和消息内容都放在主卡里明确展示

这样能第一眼回答三个问题：

- 下一条是什么？
- 发给谁？
- 什么时候送？

### 3. 忙碌态按钮不能像 debug 工具

旧版的两个彩色小方块功能上没问题，但体验太“开关化”。

新方案里：

- 主动作：`放入发件夹`
- 危险动作：`立刻插队`

主次分清楚，默认行为也更符合直觉。
