# 设置台 UI / UX 重构方案

## 现状判断

当前设置台外壳已经接入猫窝设计语言，但结构和内容还停留在“后台配置面板”阶段：

- 主容器 [SettingsPanel](/Users/wangzhao/Documents/claude_projects/catwork/web/src/components/settings/SettingsPanel.tsx:24) 用单层弹窗承载 13 个并列 tab，缺少分组和优先级
- 左侧导航把“配置”“监控”“一次性操作”“只读看板”混在一起，用户进入后要先理解分类，才能开始工作
- 内容区大量组件仍使用默认灰表格、蓝按钮、传统 CRUD 卡片，和全站 `nest-*` 语汇脱节
- 不同 tab 的任务节奏完全不同，但 UI 没有给出路径提示
  - `CatSettings` / `AccountSettings` 是实体管理
  - `CapabilityBoard` / `GovernanceSettings` 是系统健康和治理
  - `QuotaBoard` / `LeaderboardTab` 更接近只读态势页，不像“设置”

这导致设置台看起来像“套了品牌壳的后台目录树”，而不是 Cat Café 里的“工作室控制台”。

## 根因

根因不是“tab 太多”本身，而是用单一扁平导航承载了四种不同心智任务：

1. 我想配置身份与接入
2. 我想确认系统现在健不健康
3. 我想调整自动化或协作策略
4. 我只是来查看数据，不一定要改配置

F099 已经证明，异质功能不能继续塞进一维平铺导航。设置台正在重演同一个问题，只是换了容器。

## 设计目标

把设置台从“功能目录”改成“有节奏的控制台”：

1. 一眼看出哪些项目最值得先处理
2. 导航按用户意图分组，不按实现模块分组
3. 配置型页面、监控型页面、危险操作型页面有不同视觉节奏
4. 延续 F056 的暖色、圆润、克制隐喻，不退回通用 SaaS 风格
5. 移动端不再照搬桌面双栏，而是切成单列工作流

## 重构方向

### 1. 入口先给“总览”，不是直接落第一个 tab

设置台默认页改成 `概览 Overview`，顶部展示 4 张状态卡：

- 身份与账号：已绑定账号数、异常账号数、未绑定猫数
- 运行与能力：MCP/Skill 探测状态、权限异常、环境缺失
- 协作自动化：调度任务、PR 审阅、设备接入状态
- 风险与治理：治理过期项目、需同步项目、失败项

作用是让铲屎官先回答“现在先处理哪一块”，而不是一进来就在 13 个标签中猜方向。

### 2. 左侧从“tab 列表”改成“分组导航”

建议将现有内容收敛为 4 组：

| 分组 | 包含内容 | 用户意图 |
|------|----------|----------|
| 身份与接入 | 猫咪、账号、连接器 | 我要让谁可用、接到哪里 |
| 运行与策略 | 能力、权限、环境变量、外观 | 我要调整系统行为 |
| 协作与自动化 | 调度、PR 审阅、Limb 设备 | 我要让团队和工具跑起来 |
| 治理与观察 | 治理、配额、排行榜 | 我要看系统状态 |

额外决策：

- `QuotaBoard` 和 `LeaderboardTab` 继续留在设置台会稀释“设置”心智，更适合后续毕业到 Hub / 观察类入口
- 在毕业前，可以先放入“治理与观察”分组，避免与配置项并列

### 3. 内容页统一成“三段式结构”

每个设置子页统一成三段，而不是各写各的：

1. **页头**
   页面标题、简短说明、状态摘要、主操作

2. **主内容区**
   2 到 5 个 section card，每张卡只承载一类决策

3. **上下文侧栏或页内提示**
   风险提示、说明文本、最近操作、帮助文案

这样可以把“读信息”“改配置”“执行危险动作”拆开，不再让所有元素挤在同一平面。

### 4. 配置页不用表格优先，用卡片和 section 优先

当前 `GovernanceSettings` 和 `CapabilityBoard` 大量依赖表格。表格适合对比，不适合设置。

建议规则：

- 配置对象数量少于 20 时，优先卡片式 section
- 只有需要横向对比大量字段时才用表格
- 表格出现时，也要包裹在 `nest-card` 风格容器里，并加明确的上层说明

应用到现有页面：

- `CatSettings`：保留实体卡片，但增强为“身份卡 + 绑定关系 + 快捷动作”
- `AccountSettings`：账号卡片分为“连接状态”“认证方式”“绑定猫咪”“风险提示”
- `GovernanceSettings`：由裸表格改成“项目健康卡列表”，异常项目自动顶到前面
- `CapabilityBoard`：从全量矩阵表转为“默认策略 + 单猫覆盖抽屉”

## 视觉语言建议

### 1. 外层继续暖象牙，但内部不能退回默认灰蓝后台

沿用 [index.css](/Users/wangzhao/Documents/claude_projects/catwork/web/src/index.css:7) 已有 token：

- 背景仍是 `--bg-canvas / --panel / --panel-strong`
- 强调色继续用 `--accent / --accent-deep`
- 辅助状态引入更明确的语义色层，但通过猫窝色板调和

视觉原则：

- 重点不是“多装饰”，而是让每个层级的材质感清楚
- 配置区是纸卡/面板
- 风险区是轻微染色，不要直接跳回通用红框
- 只读指标是柔和块面，不要像传统运维大屏

### 2. 导航做成“分组胶囊 + 当前项高亮”

不是把所有项做成一样的按钮，而是：

- 分组标题用 `nest-kicker`
- 组内项用更紧凑的 pill row
- 当前项拥有更强的背景、阴影和左侧色条
- 异常项可带 attention dot

这样能把“结构层次”和“当前焦点”同时说清。

### 3. 强化状态语义，而不是强化装饰

每个页面都应该能显式表达四种状态：

- `healthy`：正常
- `attention`：建议处理
- `missing`：未配置 / 缺失
- `busy`：探测中 / 保存中 / 同步中

这组状态语义要在所有设置页复用，避免每个组件自己发明一套样式。

## 关键页面建议

### 概览页

- 顶部：设置台标题 + 搜索框 + “最近改动”入口
- 中部：4 张总览卡，按风险排序
- 底部：最近待处理事项列表
  - 未激活项目
  - 账号缺 API Key
  - 环境变量缺失
  - 能力探测失败

### 猫咪 / 账号页

- 把“新增”按钮降到次级位置，主焦点放在已有实体健康度
- 每张卡都能直接回答：
  - 这个对象能不能用
  - 它连接到什么
  - 缺什么
  - 下一步最可能的操作是什么

### 能力 / 权限页

- 不建议一上来给出全矩阵
- 先显示默认策略，再显示“个别覆盖”
- 个别覆盖放到抽屉或展开区，避免初始画面噪音过大

### 治理页

- 以“需要处理的项目”为主排序，而不是按路径排序
- 每个项目卡显示：
  - 状态
  - 最近同步时间
  - 主要告警
  - 推荐动作

### 外观页

- 适合做成更完整的选择卡片，而不是轻量单选框
- 这页应该成为设置台最“好看”的页面之一，因为它承担品牌感示范作用

## 交互细节

### 1. 搜索必须直达设置项，不只搜页面标题

设置搜索应该能匹配：

- 页面名
- section 名
- 核心字段名
- 常见问题关键词

例如输入“API Key”可以直接落到账号页的认证 section，而不是只高亮“账号配置”。

### 2. 危险动作和普通保存动作拆开

删除项目、重置配置、解除绑定这类危险动作，不应该和“刷新”“探测”“保存”长得一样。

建议：

- 普通动作：次级按钮
- 主动作：品牌强调按钮
- 危险动作：文本按钮或轻染色 danger button，且二次确认

### 3. 页面内部可有局部 sticky action bar

对于需要多步编辑的页面，可以在底部或顶部提供局部粘性操作条：

- 保存
- 放弃修改
- 恢复默认
- 查看变更

如果页面本身是即时保存，则明确写出“修改后自动保存”，不要让用户猜。

## 响应式策略

桌面端：

- 左侧保留分组导航
- 中间主内容区为主
- 必要时右侧轻量上下文栏

移动端：

- 设置台直接全屏，不再保留宽弹窗
- 顶部显示当前分组和返回
- 分组导航变为横向切换 + 页面内 section 折叠
- 数据密集矩阵默认折叠，仅在展开后查看

## 落地顺序

### Phase 1: 外壳重构，不改业务 API

- `SettingsPanel` 增加概览页
- 13 个 tab 收敛为分组导航
- 页头、状态摘要、搜索入口统一
- 先不深改各业务组件

### Phase 2: 优先重做 4 个高价值页面

- `CatSettings`
- `AccountSettings`
- `CapabilityBoard`
- `GovernanceSettings`

理由：这 4 页最能体现设置台的核心价值，也最能暴露当前结构问题。

### Phase 3: 毕业只读看板

- 评估 `QuotaBoard`、`LeaderboardTab` 是否迁出设置台
- 若仍保留，则用“治理与观察”模式承接

## 与参考实现对齐后的收获

参考实现里有两点值得吸收，但不能直接照搬：

1. 设置头部要显式表达“是否有未保存变更”
2. 内容页应以 section 为组织单元，而不是裸表格堆叠

我们与参考实现不同的地方是：

- 我们需要延续 Cat Café 的暖色与圆润材质，而不是中性 IDE 面板
- 我们的设置对象更复杂，既有实体管理，也有治理、自动化和团队协作
- 所以不能只做一个普通 settings dialog，必须有更强的信息分层

## 给 @opus 的对齐问题

1. `QuotaBoard` / `LeaderboardTab` 是否同意后续从设置台毕业到独立观察入口
2. `CapabilityBoard` 是否接受“默认策略 + 单猫覆盖抽屉”替代全矩阵首屏
3. 设置搜索是否可先做前端索引，后续再补后端可搜索元数据
4. 是否要引入统一的“自动保存 / 显式保存”页面契约，避免每页语义不同

## 最小实现建议

如果本轮只做一个最小闭环，我建议先做下面 3 件事：

1. `SettingsPanel` 增加 `概览` 与分组导航
2. `GovernanceSettings` 从表格改成风险排序卡片
3. `AccountSettings` 把“绑定关系 / 认证状态 / 下一步动作”做成信息更清楚的账号卡

这样能在不动太多业务逻辑的情况下，让设置台从“目录树”迈向“控制台”。
